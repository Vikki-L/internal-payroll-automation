# 03 · Edge Cases · 容错、风控与跨境分流

> 把"95% 的主流程"和"5% 的特殊情况"分开设计，是这套系统能稳定上线的关键。

## 🛡️ 容错机制

### 1. Creator 长期不确认 → 超期废弃流程

**问题**：Creator 不及时确认薪酬明细，结算挂账无限期累积——影响 Coach 进度、影响财务对账。

**解决方案 · 三阶段超期管理**：

```
   Day 0:  Coach 发起 Creator 确认通知
   ↓
   Day 1-N:  Creator 看到明细，应在窗口内确认
   ↓
   Day N1:  超期未确认 → 进入预警状态（系统自动通知）
   ↓
   Day N2:  仍未确认 → 进入"自动废弃候选"状态
            · 通知 Coach 介入
            · Coach 决定：联系 Creator / 直接废弃
   ↓
   Day N3:  仍未处理 → 系统自动废弃
            · 该笔结算金额归零
            · 后续如要重发需 Coach 主动重新发起
```

**关键设计**：
- **三阶段缓冲**（预警 → 候选 → 废弃）防止误废
- **Coach 有介入点**（在自动废弃前能介入）
- **可逆性**：被废弃的结算可以由 Coach 重新发起

### 2. 调整项必填理由

任意金额奖励 / 扣款必须填**至少 N 字理由**：

```
正例：
  调整项: +$50
  理由: "本周超额完成 quota，奖励"
  
反例（被拒）：
  调整项: +$200
  理由: "好"    ← 字数不够，禁止提交
```

**为什么**：
- 防止 Coach 滥用调整权限
- 留下审计记录（财务/合规需要）
- Creator 自助确认时能看到调整理由（避免争议）

### 3. 分级状态只读

结算模块只能**读取**分级状态，不能修改：

```
其他模块: 分级管理（评级 / 升降级流程）
   ↓
   写入分级状态
   ↓
结算模块: 只读分级状态 → 决定单价档位
```

**为什么**：
- 单一数据源（避免不同模块对分级的"理解"不一致）
- 避免越权（Coach 不能为了多结算钱给 Creator 升级）

## 🌍 Geo 分流（跨境合规）

不同国家的 Creator 走完全不同的薪酬流程：

### US Creator → 自动主流程

```
算薪 → 自助确认 → CSV → 美区薪酬平台 → 跨境打款（USD）
```

### Non-US Creator → 手动兜底流程

```
算薪 → 自助确认 → 进入"手动结算列表"
   ↓
   Coach / Finance 手动处理：
   · 走当地合规的支付通道
   · 手动登记到账状态
   · 标记结算完成
```

**为什么需要分流**：
- 美区薪酬平台只覆盖 US 主流场景
- 其他国家需要走不同合规通道（如 Wise / PayPal / 当地银行电汇）
- 合规风险高 → 自动化反而不安全

**关键设计**：
- 系统**自动识别 Geo**（按 Creator 填写的居住国）
- US 自动跑 / Non-US 自动进入手动列表
- **不让 Coach 选择走哪条流程**（避免合规风险）

## 🔐 风控机制

### 1. 异常金额警告

- 单笔结算金额超过历史 P95 → 系统标记 + 要求 Coach 二次确认
- 调整项金额超过基础结算 50% → 自动转人工审批

### 2. Creator 重复确认防滥用

- 一个结算窗口只能确认一次
- 确认后才能进入打款流程
- 异议提出后回到 Coach 重新算（避免无限循环）

### 3. CSV 导出审计留痕

- 每次 CSV 导出系统记录：操作人 / 时间 / 金额合计 / Creator 数量
- 后续打款失败可追溯到具体的导出批次

---

## 💡 Why Edge Cases Matter

**主流程能不能跑通决定产品能不能上线，边界情况能不能处理决定产品能不能用得长久**。

这套结算系统主流程很简单（算 → 确认 → 打款），但是：
- 超期不确认怎么办？
- Non-US Creator 怎么办？
- 异常金额怎么办？
- 调整项滥用怎么防？

**这些 edge case 的设计才是产品的差异化**——直接复制主流程很容易，但把边界情况想清楚是真功夫。
